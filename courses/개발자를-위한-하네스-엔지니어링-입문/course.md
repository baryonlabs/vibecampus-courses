# 하네스 엔지니어링 입문

AI를 잘 쓰는 사람과 결과를 반복 재현하는 사람의 차이

<!-- script: 이번 무료 강의에서는 하네스 엔지니어링의 핵심 개념을 잡습니다. AI에게 한 번 잘 시키는 수준이 아니라, 같은 품질의 결과를 반복 재현하는 구조를 만드는 법을 배웁니다. -->

---

# 왜 하네스가 필요한가

- 좋은 프롬프트 하나만으로는 결과 품질이 들쑥날쑥합니다.
- 사람이 바뀌어도 같은 방식으로 실행되는 작업 구조가 필요합니다.
- 입력, 제약, 검증, 보고를 묶은 운영틀이 있어야 팀 생산성이 유지됩니다.
- AI를 도구가 아니라 작업자처럼 쓰려면 하네스가 필요합니다.

<!-- script: 하네스가 필요한 이유는 단순합니다. 좋은 프롬프트 하나만으로는 결과 품질이 매번 같지 않기 때문입니다. 입력, 제약, 검증, 보고를 묶은 운영틀이 있어야 AI를 진짜 작업자처럼 쓸 수 있습니다. -->

---

# 프롬프트와 하네스의 차이

- 프롬프트: 한 번의 지시 문장
- 하네스: 지시 + 입력형식 + 실행순서 + 검증기준 + 실패시 처리
- 프롬프트는 아이디어를 주고, 하네스는 결과를 관리합니다.
- 하네스가 있어야 팀원이 바뀌어도 같은 실행 품질을 기대할 수 있습니다.

<!-- script: 프롬프트가 말 그대로 한 번의 지시라면, 하네스는 실행 전체를 감싸는 틀입니다. 입력 형식, 실행 순서, 검증 기준, 실패 처리까지 묶여 있어야 실제 운영에서 쓸 수 있습니다. -->

# 실무용 하네스의 4요소

입력, 제약, 검증, 보고

<!-- script: 실무에서 쓰는 하네스는 네 가지 요소로 단순하게 잡으면 됩니다. 입력, 제약, 검증, 보고. 이 네 가지가 빠지면 결국 사람 손으로 다시 수습하게 됩니다. -->

---

# 1. 입력을 표준화한다

- 무엇을 바꾸는지보다 왜 바꾸는지 먼저 적습니다.
- 파일 경로, 참고 문서, 기대 산출물을 명시합니다.
- 가능하면 예시 입력과 예시 출력을 같이 둡니다.

<!-- script: 첫 번째는 입력 표준화입니다. 무엇을 바꾸는지만 적지 말고 왜 바꾸는지까지 적어야 합니다. 파일 경로, 참고 문서, 기대 산출물도 같이 명시해야 AI가 덜 헤맵니다. -->

---

# 2. 제약과 승인 게이트를 건다

- 배포, 비용 증가, 외부 공개는 승인 포인트를 분리합니다.
- 수정 범위를 파일 또는 모듈 단위로 제한합니다.
- 실패 시 되돌리는 기준도 미리 정의합니다.

<!-- script: 두 번째는 제약과 승인 게이트입니다. 배포나 외부 공개처럼 중요한 분기는 자동으로 넘기지 말고 승인 포인트를 분리해야 합니다. 그래야 사고를 줄일 수 있습니다. -->

---

# 3. 검증과 보고를 붙인다

- 실행 후 최소 한 번은 재현 테스트를 강제합니다.
- 성공 여부보다 로그와 파일 경로를 같이 남깁니다.
- 다음 액션은 15~30분 단위로 잘게 제안합니다.

<!-- script: 세 번째는 검증과 보고입니다. 성공했다고 말하는 것만으로는 부족합니다. 실제 로그, 파일 경로, 검증 결과까지 남겨야 다음 사람이 이어받을 수 있습니다. -->

# 무료 실습 순서

- 기능 하나를 고릅니다. 예: 랜딩 CTA 문구 수정
- 목표, 파일 경로, 금지사항, 완료 기준을 5줄 안에 적습니다.
- AI 에이전트에게 실행시키고 변경 로그를 받습니다.
- 직접 클릭해서 검증하고, 다음 액션 1개만 남깁니다.

<!-- script: 실습은 간단합니다. 기능 하나를 고르고, 목표와 파일 경로, 금지사항, 완료 기준을 짧게 적습니다. 그 다음 AI에게 실행시키고, 직접 검증한 뒤, 다음 액션 한 개만 남기면 됩니다. -->

# 이 강의 후 바로 얻게 되는 것

- AI에게 일을 잘 맡기는 기준선
- 결과 품질을 흔들리지 않게 만드는 운영 습관
- 팀/개인 프로젝트에 바로 적용 가능한 미니 하네스 템플릿

<!-- script: 이 강의가 끝나면 AI에게 일을 잘 맡기는 기준선이 생깁니다. 결과 품질을 흔들리지 않게 만드는 운영 습관도 함께 가져갈 수 있습니다. -->
