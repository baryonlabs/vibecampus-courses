---
# Level 4 · 서브 에이전트 팀

한 명짜리 비서에서 팀으로 넘어가야 하는 이유를 알아봅니다.

---

# Level 3까지의 성과

- 프롬프트로 AI를 부릴 수 있다
- 도구(검색, 코드 실행)를 연결했다
- 하나의 에이전트가 꽤 많은 일을 한다

---

# 그런데 왜 부족한가?

- 긴 작업에서 맥락이 흐려진다
- 조사와 작성을 동시에 시키면 품질이 떨어진다
- 자기 결과를 자기가 검증하면 실수를 못 잡는다
- 역할이 섞이면 프롬프트가 비대해진다

---

# 인간 조직과 비교

- 기자 한 명이 취재·작성·교정을 다 하면?
- 품질 저하, 피로, 실수 증가
- AI도 마찬가지 — 역할 집중이 답이다

---

# 단일 에이전트 실패 패턴

- 환각 누적: 조사 결과를 자기가 만들어낸 정보로 덮어씀
- 지시 충돌: '자세히 써라'와 '간결하게 검토하라'가 한 프롬프트에 공존
- 디버깅 불가: 어디서 잘못됐는지 추적이 안 됨

---

# 이 레슨 핵심 정리

- 복잡한 작업은 단일 에이전트로 한계가 있다
- 역할 분리만으로도 품질이 크게 달라진다
- 다음 레슨에서 역할 분리 개념을 구체화한다

---

<!-- +lesson: 2 | 역할 분리라는 개념 -->

# 역할 분리 = 관심사 분리

소프트웨어 설계 원칙을 AI 팀에 적용합니다.

---

# 역할 분리란?

- 하나의 큰 작업을 독립된 책임 단위로 나누는 것
- 각 역할은 명확한 입력을 받고 명확한 출력을 내보냄
- 역할 간 인터페이스(계약)가 존재

---

# 대표 역할 3가지

- 조사자(Researcher): 주어진 질문에 대해 사실 기반 정보를 수집
- 작성자(Writer): 조사 결과를 바탕으로 초안 작성
- 검토자(Reviewer): 초안을 사실 확인·톤·완성도 기준으로 평가

---

# 왜 세 역할인가?

- 조사 → 생성 → 검증은 지식 작업의 최소 파이프라인
- 각 단계의 성공 기준이 다르다
- 검증 단계가 빠지면 환각을 잡을 수 없다

---

# 역할 분리의 즉각적 이점

- 프롬프트가 짧아지고 집중된다
- 각 단계를 독립적으로 테스트할 수 있다
- 실패 지점을 빠르게 특정할 수 있다

---

# 레슨 정리

- 역할 분리 = 관심사 분리
- 조사·작성·검토가 최소 구성
- 다음 레슨에서 실제 사례를 본다

---

<!-- +qa -->
- q: "역할을 더 세분화할 수도 있나요?"
  a: "가능합니다. 예를 들어 '팩트체커'와 '편집자'를 분리할 수 있습니다. 다만 역할이 많아지면 통신 비용이 늘어나므로 최소한으로 시작하는 것이 좋습니다."
- q: "역할을 나누면 비용이 더 드나요?"
  a: "API 호출 수는 늘어나지만, 각 호출의 프롬프트가 짧아져 전체 토큰 사용량은 비슷하거나 오히려 줄기도 합니다. 품질 대비 비용 효율은 높아집니다."

# 실제로 팀이 동작하는 예시

이론이 아닌 실전 사례로 감을 잡습니다.

---

# 사례 1 — 경쟁사 분석 보고서

- 조사자: 경쟁사 3곳의 최근 뉴스·IR 자료 수집
- 작성자: 요약 보고서 초안 작성
- 검토자: 출처 누락·수치 오류 검증
- 결과: 단일 에이전트 대비 사실 정확도 40% 향상

---

# 사례 2 — 기술 블로그 작성

- 조사자: 관련 논문·공식 문서 요약
- 작성자: 블로그 글 초안 생성
- 검토자: 기술 정확성·가독성 점검
- 결과: 수정 반복 횟수가 3회에서 1회로 줄어듬

---

# 사례 3 — 고객 문의 대응

- 조사자: FAQ·정책 문서에서 관련 항목 검색
- 작성자: 고객 응답 초안 생성
- 검토자: 정책 준수 여부·톤 확인
- 결과: 응답 정확도와 일관성이 동시에 개선

---

# 공통 패턴 정리

- 세 사례 모두 조사→작성→검토 순서
- 검토자가 있어야 품질 보증이 가능
- 역할별 프롬프트가 짧고 명확

# 에이전트 팀 구조

*조사자 + 작성자 + 검토자 → 고품질 결과물*

---
# 역할 정의 실습

각 에이전트의 시스템 프롬프트와 책임 범위를 정합니다.

---

# 좋은 역할 정의의 조건

- 이름과 한 줄 설명이 명확하다
- 할 수 있는 일과 하면 안 되는 일이 구분된다
- 사용할 도구가 지정된다
- 성공 기준이 측정 가능하다

---

# 조사자(Researcher) 정의 예시

- 이름: researcher
- 역할: 주어진 질문에 대해 신뢰할 수 있는 정보를 수집한다
- 도구: 웹 검색, 문서 검색
- 금지: 자기 의견을 섞지 않는다
- 출력: 출처가 명시된 요약 목록

---

# 작성자(Writer) 정의 예시

- 이름: writer
- 역할: 조사 결과를 바탕으로 지정된 형식의 문서를 작성한다
- 도구: 없음 (입력 자료만 사용)
- 금지: 조사 결과에 없는 내용을 추가하지 않는다
- 출력: 마크다운 초안

---

# 검토자(Reviewer) 정의 예시

- 이름: reviewer
- 역할: 초안을 사실 정확성·완성도·톤 기준으로 평가한다
- 도구: 조사 결과 원문 참조
- 금지: 직접 수정하지 않는다 — 피드백만 제공
- 출력: 통과/재작성 판정 + 구체적 피드백

---

# 실습 과제

- 자신의 업무 하나를 골라 세 역할로 정의해 보세요
- 각 역할의 이름·책임·도구·금지사항·출력을 적어 보세요
- 역할 간 겹치는 부분이 없는지 확인하세요

---

# 레슨 정리

- 역할 정의는 시스템 프롬프트의 골격이다
- 이름·책임·도구·금지·출력 다섯 항목을 채우면 된다
- 다음 레슨에서 입출력 계약을 만든다

---

<!-- +qa -->
- q: "역할 정의를 너무 상세하게 쓰면 오히려 안 좋은가요?"
  a: "핵심 책임과 금지 사항만 간결하게 적는 것이 좋습니다. 지나치게 길면 모델이 주요 지시를 놓칠 수 있습니다. 5-8문장이 적당합니다."
- q: "검토자가 직접 수정하면 안 되는 이유는?"
  a: "검토자가 수정까지 하면 역할이 작성자와 섞여서 검증 기능이 약해집니다. 피드백만 주고 작성자가 반영하게 하면 각자의 전문성이 유지됩니다."

# 에이전트 간 계약

역할 사이를 오가는 데이터의 형태를 정합니다.

---

# 왜 계약이 필요한가

- 역할만 정하면 출력 형식이 제각각이다
- 다음 에이전트가 파싱에 실패하면 파이프라인이 멈춘다
- 계약 = 에이전트 간 API 스펙

---

# 계약의 구성 요소

- 입력 스키마: 어떤 필드가 필수인가
- 출력 스키마: 어떤 형식으로 내보내는가
- 제약 조건: 최대 길이, 필수 포함 항목 등

---

# 조사자 → 작성자 계약 예시

- 조사자 출력: { findings: [{ title, summary, source_url }] }
- 작성자 입력: findings 배열을 받아 마크다운 문서 생성
- 제약: findings는 3개 이상, 각 항목에 source_url 필수

---

# 작성자 → 검토자 계약 예시

- 작성자 출력: { draft: string, word_count: number }
- 검토자 입력: draft 텍스트 + 원본 findings
- 검토자 출력: { verdict: 'pass' | 'revise', feedback: string[] }

---

# 계약 위반 처리

- 출력이 스키마에 맞지 않으면 재시도
- 최대 재시도 횟수 초과 시 에러 반환
- 계약은 테스트 가능해야 한다 — 유닛 테스트처럼

---

# 레슨 정리

- 계약 = 입력 스키마 + 출력 스키마 + 제약 조건
- 계약이 명확하면 에이전트를 독립적으로 테스트 가능
- 다음 레슨에서 통신 패턴을 배운다

---

<!-- +qa -->
- q: "JSON 스키마를 꼭 써야 하나요?"
  a: "꼭 JSON일 필요는 없지만, 구조화된 형식이면 파싱과 검증이 쉬워집니다. JSON 스키마, Pydantic 모델, TypeScript 인터페이스 등 어떤 형식이든 괜찮습니다."
- q: "계약을 프롬프트에 어떻게 넣나요?"
  a: "시스템 프롬프트에 '반드시 다음 JSON 형식으로 출력하라'고 명시하고 예시를 첨부합니다. OpenAI의 JSON 모드나 function calling을 활용하면 더 안정적입니다."

# 에이전트 간 통신 패턴

순차, 병렬, 피드백 루프 — 언제 어떤 패턴을 쓸까요.

---

# 패턴 1 — 순차 파이프라인

- 조사자 → 작성자 → 검토자 순서로 실행
- 가장 단순하고 디버깅이 쉽다
- 적합: 단계 간 의존성이 강한 작업

---

# 패턴 2 — 병렬 수집 후 합류

- 조사자 A, B, C가 동시에 다른 소스를 조사
- 결과를 합쳐서 작성자에게 전달
- 적합: 독립된 정보 소스가 여러 개인 경우

---

# 패턴 3 — 피드백 루프

- 검토자가 '재작성' 판정 시 작성자에게 되돌림
- 최대 N회 반복 후 최종 결과 확정
- 적합: 품질 기준이 높은 문서 작성

---

# 패턴 선택 기준

- 작업이 단순하면 순차로 시작
- 정보 소스가 독립적이면 병렬 수집
- 품질이 중요하면 피드백 루프 추가
- 복잡도는 최소한으로 유지

---

# Stage 2 요약

- 역할 정의: 이름·책임·도구·금지·출력
- 입출력 계약: 스키마 + 제약 조건
- 통신 패턴: 순차, 병렬, 피드백 루프
- 다음 Stage에서 실제로 만들어 본다

---

<!-- +qa -->

  a: "자주 그렇게 합니다. 예를 들어 병렬 조사 → 순차 작성 → 피드백 루프 검토 조합이 일반적입니다."
- q: "AutoGen과 CrewAI 중 어떤 걸 쓰면 좋나요?"
  a: "AutoGen은 대화 기반 협업에 강하고, CrewAI는 역할 기반 워크플로 정의가 직관적입니다. 간단한 파이프라인은 CrewAI, 복잡한 대화형 협업은 AutoGen이 적합합니다."

# 에이전트 팀 구조

![에이전트 팀 구조](https://www.vibecamp.us/course-images/level4-agent-team.svg)

*조사자 + 작성자 + 검토자 → 고품질 결과물*

---
# 파이프라인 구축 실습

Stage 2에서 설계한 것을 실제로 만들어 봅니다.

---

# 구축 순서

- 1. 각 역할의 시스템 프롬프트 작성
- 2. 입출력 스키마를 코드로 정의
- 3. 순차 파이프라인으로 연결
- 4. 테스트 질문으로 실행
- 5. 각 단계 출력 확인

---

# 조사자 프롬프트 핵심

- '너는 조사자다. 주어진 질문에 대해 신뢰할 수 있는 정보만 수집한다.'
- 출력 형식을 JSON으로 고정
- 출처 URL을 반드시 포함하라고 명시

---

# 작성자 프롬프트 핵심

- '너는 작성자다. 조사 결과만을 바탕으로 문서를 작성한다.'
- 조사 결과에 없는 내용 추가 금지
- 지정된 톤과 분량 준수

---

# 검토자 프롬프트 핵심

- '너는 검토자다. 초안을 다음 기준으로 평가한다.'
- 평가 기준: 사실 정확성, 완성도, 톤 일관성
- 판정을 pass 또는 revise로 출력

---

# 연결하기

- 조사자 출력 → 작성자 입력으로 전달
- 작성자 출력 + 조사 결과 → 검토자 입력
- 검토자가 revise → 작성자 재실행 (최대 2회)

---

# 레슨 정리

- 시스템 프롬프트 3개 + 스키마 정의 + 연결 로직이 전부
- 프레임워크 없이도 API 호출 체이닝으로 구현 가능
- 다음 레슨에서 결과를 평가한다

---

<!-- +qa -->
- q: "프레임워크 없이 구현하면 코드가 많이 복잡한가요?"
  a: "순차 파이프라인이라면 함수 3개와 메인 루프 하나면 됩니다. 50줄 이내로 가능합니다. 프레임워크는 병렬 실행이나 상태 관리가 필요할 때 도입하세요."
- q: "어떤 모델을 역할별로 쓰면 좋나요?"
  a: "조사자는 도구 호출이 안정적인 모델, 작성자는 글 품질이 좋은 모델, 검토자는 판단력이 좋은 모델을 쓸 수 있습니다. 같은 모델이어도 역할 분리 효과는 충분합니다."

<!-- +lesson: 2 | 멀티에이전트 파이프라인 평가 -->

# 파이프라인 평가하기

만들었으면 측정해야 합니다.

---

# 무엇을 측정하나

- 사실 정확도: 출처와 대조하여 오류 비율 측정
- 완성도: 요구 항목이 모두 포함되었는지
- 일관성: 같은 입력에 대해 비슷한 품질의 출력
- 비용: 총 토큰 사용량과 API 호출 횟수
- 소요 시간: 전체 파이프라인 완료 시간

---

# 단일 에이전트와 비교

- 같은 질문을 단일 에이전트로도 실행
- 두 결과를 사실 정확도·완성도 기준으로 비교
- 비용 대비 품질 향상 폭을 확인

---

# 자주 발견되는 문제

- 조사자가 관련 없는 정보를 가져옴 → 질문 구체화 필요
- 작성자가 조사 결과를 무시 → 프롬프트에 '반드시 참조' 강조
- 검토자가 항상 pass → 검토 기준을 더 구체적으로 설정

---

# 개선 루프

- 문제 발견 → 해당 역할의 프롬프트 수정
- 수정 후 동일 테스트로 재실행
- 개선 전후 점수 비교
- 이 과정을 체계화하면 Level 5 하네스 영역

---

# 레슨 정리

- 측정 없이는 개선할 수 없다
- 단일 에이전트 대비 비교가 핵심
- 문제 진단 → 프롬프트 수정 → 재측정 루프를 반복
- 다음 레슨에서 한계와 Level 5 필요성을 정리한다

---

<!-- +qa -->
- q: "평가를 자동화할 수 있나요?"
  a: "가능합니다. 예를 들어 '별도의 평가 에이전트'가 정답 기준과 비교해서 점수를 매기게 할 수 있습니다. 하지만 초기에는 사람이 직접 확인하는 것이 정확합니다."
- q: "비용이 2배 가까이 드는데 괜찮은가요?"
  a: "품질이 크게 향상되면 수작업 수정 시간이 줄어들어 총비용은 오히려 낮아질 수 있습니다. 비용 대비 가치(ROI)로 판단하세요."

# Level 4의 한계

팀은 있지만 운영 체계가 부족합니다.

---

# 지금까지 배운 것

- 역할을 분리하면 품질이 올라간다
- 입출력 계약으로 안정적인 파이프라인을 만든다
- 평가와 개선 루프로 점진적으로 발전한다

---

# 하지만 여전히 남는 문제

- 에이전트 팀 구성을 매번 수동으로 설정해야 한다
- 로그가 흩어져서 무엇이 잘못됐는지 추적이 어렵다
- 재시도 정책, 타임아웃, 에러 핸들링이 일관되지 않다
- 여러 파이프라인을 관리하면 복잡도가 폭증한다

---

# Level 5 하네스가 해결하는 것

- 에이전트 팀의 선언적 정의 — 코드가 아니라 설정으로
- 중앙 집중 로깅 — 모든 에이전트의 입출력을 한 곳에서 확인
- 재시도·타임아웃·폴백 정책의 표준화
- 파이프라인의 버전 관리와 재현 가능성

---

# Level 4와 Level 5의 차이 비유

- Level 4: 개발자가 스크립트로 서버를 띄우는 단계
- Level 5: Kubernetes처럼 오케스트레이션이 있는 단계
- 팀은 같지만 운영 방식이 근본적으로 다르다

---

# 전체 과정 요약

- Stage 1: 왜 팀이 필요한가 — 단일 에이전트의 한계
- Stage 2: 역할·계약·통신 패턴 설계
- Stage 3: 파이프라인 구축·평가·한계 인식
- 다음 단계: Level 5 하네스로 운영 체계를 갖추자
