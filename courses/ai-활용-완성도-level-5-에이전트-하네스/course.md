# 하네스 시스템

![하네스 시스템](https://www.vibecamp.us/course-images/level5-harness.svg)

*Ad-hoc 팀 vs 운영 가능한 시스템 비교*

---
# Stage 1 · 팀에서 시스템으로

잘 돌아가는 것과 반복 가능한 것은 완전히 다릅니다.

---

# 애드혹 팀 vs 시스템

- 애드혹: 사람이 매번 판단하고 조율한다
- 시스템: 입력이 들어오면 정해진 경로를 따라 결과가 나온다
- 핵심 차이는 '사람 없이도 같은 결과가 나오는가'이다

![Level 5 개념도](https://www.vibecamp.us/course-images/level5-harness.svg)

---

# 시스템의 4가지 조건

- 명확한 입력과 출력 정의
- 실행 경로가 문서화되어 있다
- 실패했을 때 어떻게 되는지 정해져 있다
- 누가 실행해도 같은 결과를 기대할 수 있다

---

# 에이전트 팀이 시스템이 되려면

- 각 에이전트의 역할 경계가 명확해야 한다
- 에이전트 간 데이터 전달 형식이 고정되어야 한다
- 전체 흐름을 한눈에 볼 수 있어야 한다

---

# 실습: 내 워크플로 점검

- Level 4에서 만든 에이전트 팀을 꺼내기
- 각 단계를 '입력 → 처리 → 출력' 형태로 정리
- 사람이 개입하는 지점 표시하기
- 그 지점을 자동화하면 무엇이 필요한지 적기

---

# 핵심 정리

- 시스템 = 반복 가능 + 재현 가능 + 관찰 가능
- 에이전트를 '잘 쓰는 것'과 '운영하는 것'은 다른 스킬이다
- 이 강의의 목표는 운영 가능한 구조를 만드는 것이다

---

<!-- +qa -->
- q: "에이전트 2~3개만 쓰는데도 시스템이 필요한가요?"
  a: "에이전트 수가 아니라 반복 횟수가 기준입니다. 같은 작업을 주 3회 이상 한다면 시스템화 가치가 충분합니다."
- q: "시스템화하면 유연성이 떨어지지 않나요?"
  a: "좋은 시스템은 변경 지점을 명시합니다. 전체를 고정하는 게 아니라, 바뀌는 부분과 고정하는 부분을 분리하는 것입니다."

# 상태 관리

에이전트가 지금 어디에 있는지 항상 알 수 있어야 합니다.

---

# 상태란 무엇인가

- 현재 어떤 단계를 실행 중인지
- 이전 단계의 결과가 무엇이었는지
- 다음에 무엇을 해야 하는지
- 이 세 가지를 알 수 있으면 '상태가 관리되고 있다'

---

# 상태가 없으면 생기는 문제

- 중간에 멈추면 처음부터 다시 해야 한다
- 어디서 실패했는지 알 수 없다
- 같은 작업을 중복 실행한다
- 디버깅이 불가능하다

---

# 상태 관리 방법

- 간단한 방법: JSON 파일에 단계별 결과 저장
- 중간 방법: 키-값 저장소(Redis 등) 활용
- 고급 방법: LangGraph의 StateGraph 활용
- 원칙: 각 단계 완료 시 상태를 반드시 기록한다

---

# 실습: 상태 스키마 만들기

- 워크플로의 각 단계를 나열한다
- 각 단계의 입력/출력을 정의한다
- 상태 객체의 형태를 JSON으로 작성한다
- 어떤 시점에서든 상태만 보면 현황을 파악할 수 있는지 확인한다

---

# 핵심 정리

- 상태 관리는 하네스의 가장 기본적인 요소이다
- 상태가 있어야 재시도, 로그, 모니터링이 가능해진다

---

<!-- +qa -->
- q: "상태를 어디에 저장하는 게 좋나요?"
  a: "프로토타입은 로컬 JSON 파일로 충분합니다. 운영 환경에서는 Redis나 DB를 쓰되, 중요한 건 저장 위치가 아니라 '매 단계마다 기록한다'는 습관입니다."
- q: "LangGraph를 꼭 써야 하나요?"
  a: "아닙니다. 개념을 이해하면 어떤 도구든 적용 가능합니다. LangGraph는 참고 구현일 뿐입니다."

# 로그와 재시도

실패는 당연합니다. 대응 방법이 설계되어 있느냐가 차이입니다.

---

# 왜 로그가 필요한가

- 에이전트는 비결정적이다 — 같은 입력에도 다른 출력이 나올 수 있다
- 문제가 생겼을 때 원인을 추적하려면 기록이 있어야 한다
- 품질 개선의 근거 데이터가 된다

---

# 무엇을 기록해야 하는가

- 각 단계의 시작/종료 시각
- 입력값과 출력값(또는 요약)
- 사용한 모델과 파라미터
- 에러 발생 시 에러 메시지와 컨텍스트
- 토큰 사용량과 비용

---

# 재시도 전략

- 즉시 재시도: 일시적 오류(네트워크, 429)에 적합
- 지수 백오프: 부하 관련 오류에 적합
- 대체 경로: 다른 모델이나 다른 프롬프트로 전환
- 인간 에스컬레이션: 자동 복구가 불가능할 때 사람에게 알림

---

# 실패 분류 체계

- 일시적 오류: 재시도하면 해결 (네트워크, 타임아웃)
- 입력 오류: 재시도해도 같은 결과 (잘못된 프롬프트)
- 시스템 오류: 외부 서비스 장애 (API 다운)
- 로직 오류: 설계 자체의 문제 (잘못된 흐름)

---

# 실습: 재시도 정책 작성

- 워크플로의 각 단계에서 발생할 수 있는 실패를 나열한다
- 각 실패를 위 4가지로 분류한다
- 분류별 대응 방법을 정한다
- 최대 재시도 횟수와 타임아웃을 설정한다

---

# 핵심 정리

- 로그 없는 에이전트 시스템은 블랙박스다
- 재시도 전략은 실패 유형별로 다르게 설계해야 한다
- 로그 + 재시도 + 상태 관리가 합쳐지면 운영 가능한 시스템의 토대가 된다

---

<!-- +qa -->
- q: "로그를 너무 많이 남기면 비용이 걱정됩니다."
  a: "핵심 정보만 구조화하여 기록하면 됩니다. 입력/출력 전문 대신 요약, 토큰 수, 에러 여부 정도만 남겨도 디버깅에 충분합니다."
- q: "재시도를 몇 번까지 하는 게 적당한가요?"
  a: "일반적으로 3회가 기본입니다. 그 이상은 근본 원인을 해결해야 하는 신호입니다. 재시도 사이 간격을 점점 늘리는 지수 백오프를 함께 사용하세요."
- q: "에러 알림은 어떻게 보내나요?"
  a: "가장 간단한 방법은 Slack 웹훅입니다. 실패 시 웹훅으로 메시지를 보내면 즉시 확인할 수 있습니다."

---
# 체크포인트와 승인 게이트

멈출 수 있어야 제어할 수 있습니다.

---

# 체크포인트란

- 워크플로 중간에 상태를 저장하는 지점
- 체크포인트가 있으면 실패 시 그 지점부터 재개 가능
- 전체를 다시 실행하지 않아도 되므로 비용과 시간을 절약

![Level 5 개념도](https://www.vibecamp.us/course-images/level5-harness.svg)

---

# 승인 게이트란

- 자동 실행을 일시 정지하고 사람의 확인을 받는 지점
- 비용이 큰 작업 전에 배치한다 (외부 API 호출, 이메일 발송 등)
- 승인/거부/수정 세 가지 경로를 설계한다

---

# 어디에 배치하는가

- 비가역적 작업 직전 (데이터 삭제, 외부 전송)
- 비용이 높은 작업 직전 (대량 API 호출)
- 품질이 중요한 결과물 생성 직후 (보고서, 고객 응대)
- 단계 간 데이터 형식이 바뀌는 지점

---

# 구현 패턴

- 동기 승인: 슬랙 버튼 클릭 후 계속 진행
- 비동기 승인: 이메일로 링크를 보내고 클릭 대기
- 자동 승인: 특정 조건(신뢰도 90% 이상)을 만족하면 자동 통과
- 시간 제한: N분 이내 응답 없으면 기본 경로 실행

---

# 실습: 체크포인트 맵 그리기

- 자신의 워크플로를 단계별로 나열한다
- 각 단계 사이에 체크포인트가 필요한지 판단한다
- 승인 게이트가 필요한 지점을 표시한다
- 각 게이트의 승인 방식(동기/비동기/자동)을 정한다

---

<!-- +qa -->
- q: "승인 게이트가 많으면 자동화 의미가 없지 않나요?"
  a: "처음에는 많이 두고, 신뢰가 쌓이면 자동 승인 조건을 넓혀가는 것이 안전합니다. 점진적으로 줄여나가는 게 핵심입니다."
- q: "체크포인트 저장에 추가 비용이 드나요?"
  a: "상태를 JSON으로 저장하는 수준이라 비용은 거의 없습니다. 체크포인트 없이 전체를 재실행하는 비용이 훨씬 큽니다."

# 에러 처리와 폴백

에이전트가 실패하는 건 버그가 아니라 정상입니다.

---

# 에이전트 시스템의 에러 특성

- LLM 응답은 비결정적이므로 예상 못한 형식이 나올 수 있다
- 외부 API는 언제든 실패할 수 있다
- 토큰 제한 초과, 레이트 리밋 등 자원 제약이 있다
- 에러를 예외가 아닌 정상 경로로 설계해야 한다

---

# 출력 검증 패턴

- 스키마 검증: 에이전트 출력이 정해진 JSON 스키마를 따르는지 확인
- 범위 검증: 숫자 값이 합리적 범위에 있는지 확인
- 일관성 검증: 이전 단계 결과와 모순되지 않는지 확인
- 검증 실패 시 재생성을 요청하거나 폴백 경로로 전환

---

# 폴백 전략

- 모델 폴백: GPT-4가 실패하면 Claude로 전환
- 프롬프트 폴백: 복잡한 프롬프트가 실패하면 단순화된 버전 사용
- 수동 폴백: 자동 처리 불가 시 사람에게 전달
- 캐시 폴백: 이전에 성공한 유사 결과를 임시로 사용

---

# 에러 전파와 격리

- 한 에이전트의 실패가 전체 시스템을 멈추면 안 된다
- 각 에이전트를 독립적으로 실패/복구할 수 있게 격리한다
- 에러 영향 범위를 제한하는 서킷 브레이커 패턴 적용
- 연쇄 실패를 막기 위한 타임아웃 설정

---

# 실습: 에러 처리 명세 작성

- 워크플로의 각 에이전트별 예상 에러를 3개 이상 나열
- 각 에러에 대한 폴백 경로를 정의
- 어떤 에러가 전체 중단을 유발하는지 표시
- 에러 알림 방법과 대상을 정한다

---

# 핵심 정리

- 에러 처리는 부가 기능이 아니라 하네스의 핵심이다
- 모든 에이전트 출력은 검증을 거쳐야 한다
- 폴백 경로가 없는 하네스는 불완전하다

---

<!-- +qa -->
- q: "모든 에러를 다 대비해야 하나요?"
  a: "빈도와 영향도를 기준으로 우선순위를 매기세요. 자주 발생하거나 치명적인 에러부터 처리하고, 나머지는 '수동 폴백'으로 묶어두면 됩니다."
- q: "서킷 브레이커가 뭔가요?"
  a: "연속 실패가 N회를 넘으면 해당 경로를 일시적으로 차단하는 패턴입니다. 무한 재시도를 막고 시스템 전체를 보호합니다."
- q: "출력 검증에 LLM을 또 쓰면 비용이 두 배 아닌가요?"
  a: "검증용 호출은 짧은 프롬프트로 충분하므로 비용이 크지 않습니다. JSON 스키마 검증처럼 코드로 할 수 있는 부분은 LLM 없이 처리하세요."

# 하네스 스펙 작성

지금까지 배운 것을 하나의 문서로 통합합니다.

---

# 하네스 스펙이란

- 에이전트 시스템의 설계도이자 운영 매뉴얼
- 누가 읽어도 시스템을 이해하고 실행할 수 있어야 한다
- 코드가 아니라 구조와 규칙을 정의하는 문서

---

# 스펙에 포함할 항목

- 시스템 목적과 입출력 정의
- 에이전트 목록과 각 에이전트의 역할
- 단계별 흐름과 데이터 전달 형식
- 체크포인트와 승인 게이트 위치
- 에러 처리 정책과 폴백 경로
- 로그 정책과 모니터링 항목

---

# 좋은 스펙의 조건

- 다른 사람이 이 문서만 보고 시스템을 구축할 수 있다
- 각 항목이 구체적이다 (예: '적절히 처리' → '3회 재시도 후 슬랙 알림')
- 변경 이력이 관리된다
- 실제 운영과 문서가 일치한다

---

# 스펙 작성 순서

- 1. 전체 목적을 한 문장으로 정의
- 2. 입력과 최종 출력을 명시
- 3. 중간 단계를 나열하고 순서를 정한다
- 4. 각 단계의 에이전트와 도구를 배정
- 5. 체크포인트, 승인, 에러 처리를 추가
- 6. 로그와 모니터링 항목을 정의

---

# 실습: 나만의 하네스 스펙 만들기

- 실제 반복하는 업무 하나를 선택한다
- 위 순서를 따라 스펙 문서를 작성한다
- 동료에게 보여주고 이해 가능한지 확인한다
- harness-100 저장소의 예시와 비교한다

---

# 핵심 정리

- 스펙은 코드보다 먼저 존재해야 한다
- 좋은 스펙이 있으면 구현은 도구에 상관없이 가능하다
- 스펙 작성 능력이 하네스 엔지니어링의 핵심 역량이다

# 출력 모니터링

돌아가는 것과 잘 돌아가는 것은 다릅니다.

---

# 왜 모니터링이 필요한가

- 에이전트 출력 품질은 시간이 지나면 변할 수 있다 (모델 업데이트, 데이터 변화)
- 비용이 예상을 초과할 수 있다
- 사용자 불만이 쌓이기 전에 문제를 발견해야 한다

![Level 5 개념도](https://www.vibecamp.us/course-images/level5-harness.svg)

---

# 모니터링 대상

- 성공률: 전체 실행 중 정상 완료 비율
- 품질 점수: 출력물의 정확도/관련성 (자동 평가 또는 샘플링)
- 지연 시간: 각 단계와 전체 소요 시간
- 비용: 토큰 사용량과 API 호출 비용
- 에러 패턴: 반복되는 실패 유형

---

# 자동 품질 평가 방법

- 규칙 기반: 출력 길이, 필수 키워드 포함 여부
- LLM 기반: 별도 모델로 출력을 평가 (채점 프롬프트)
- 비교 기반: 이전 성공 결과와 유사도 측정
- 사용자 피드백: 최종 사용자의 만족도 수집

---

# 대시보드 구성 예시

- 일별 실행 횟수와 성공률 추이
- 평균 소요 시간과 비용 추이
- 에러 유형별 발생 빈도
- 품질 점수 분포
- 간단한 스프레드시트로도 시작 가능

---

# 실습: 모니터링 항목 정의

- 자신의 하네스 스펙에서 모니터링할 지표 5개를 선정
- 각 지표의 정상 범위와 알림 임계값을 정한다
- 주간 리뷰 체크리스트를 만든다

---

# 핵심 정리

- 모니터링 없는 운영은 눈 감고 운전하는 것과 같다
- 완벽한 대시보드보다 핵심 지표 3~5개를 꾸준히 보는 것이 중요하다

---

# 피드백 루프

좋은 시스템은 쓸수록 나아집니다.

---

# 피드백 루프란

- 출력 결과를 평가하고 그 평가를 다시 시스템에 반영하는 순환 구조
- 수동 루프: 사람이 결과를 보고 프롬프트나 설정을 수정
- 자동 루프: 시스템이 평가 결과를 바탕으로 스스로 조정

---

# 수동 피드백 루프 설계

- 주간 리뷰에서 실패 사례를 분석한다
- 실패 원인을 분류하고 개선 방안을 정한다
- 프롬프트 수정, 흐름 변경, 에러 처리 보강 등을 적용한다
- 개선 전후 지표를 비교한다

---

# 자동 피드백 루프 패턴

- 프롬프트 자동 튜닝: 성공/실패 사례를 few-shot 예시로 자동 추가
- 경로 자동 선택: 입력 유형에 따라 최적 에이전트 경로를 학습
- 임계값 자동 조정: 승인 게이트의 자동 통과 기준을 성과에 따라 조정
- 주의: 자동 루프는 검증 없이 적용하면 품질이 급격히 떨어질 수 있다

---

# 개선 기록 관리

- 모든 변경을 버전으로 관리한다 (프롬프트도 버전 관리 대상)
- 변경 이유와 기대 효과를 기록한다
- A/B 테스트로 개선 효과를 검증한다
- 롤백이 가능하도록 이전 버전을 보존한다

---

# 실습: 피드백 루프 설계

- 하네스의 약한 지점(가장 자주 실패하는 단계)을 찾는다
- 그 지점에 수동 피드백 루프를 설계한다
- 어떤 조건에서 자동 피드백으로 전환할 수 있는지 기준을 정한다

---

# 핵심 정리

- 피드백 루프가 없으면 시스템은 정체된다
- 수동 → 반자동 → 자동 순으로 점진적으로 발전시킨다
- 자동 루프에는 반드시 안전장치(검증, 롤백)가 필요하다

---

<!-- +qa -->
- q: "프롬프트 버전 관리는 어떻게 하나요?"
  a: "가장 간단한 방법은 Git으로 프롬프트 파일을 관리하는 것입니다. 변경할 때마다 커밋 메시지에 이유를 적으세요."

  a: "자동 변경에 항상 상한선을 두세요. 예를 들어 '프롬프트 길이가 기존의 2배를 넘으면 적용 중단' 같은 규칙을 정합니다."
- q: "소규모 팀에서도 A/B 테스트가 가능한가요?"
  a: "정식 A/B 테스트가 아니어도 됩니다. 이전 버전과 새 버전을 같은 입력에 돌려보고 결과를 비교하는 것만으로 충분합니다."

# 구축 vs 구매

모든 것을 직접 만들 필요는 없습니다.

---

# 구축이 적합한 경우

- 우리 업무에 특화된 로직이 핵심일 때
- 데이터 보안이 중요하여 외부 서비스를 쓸 수 없을 때
- 기존 도구가 우리 워크플로에 맞지 않을 때
- 장기적으로 자체 역량을 축적해야 할 때

---

# 구매(외부 도구)가 적합한 경우

- 표준적인 작업이어서 이미 좋은 솔루션이 있을 때
- 빠르게 검증하고 싶을 때 (프로토타입 단계)
- 유지보수 인력이 부족할 때
- 규모가 커서 인프라 관리가 부담될 때

---

# 판단 프레임워크

- 1. 이 기능이 우리의 핵심 차별점인가? → Yes면 구축
- 2. 기존 도구로 80% 이상 해결 가능한가? → Yes면 구매
- 3. 유지보수할 여력이 있는가? → No면 구매
- 4. 6개월 후에도 이 방식을 쓸 것인가? → No면 구매로 검증 후 판단

---

# 하이브리드 전략

- 핵심 오케스트레이션은 직접 구축
- 개별 에이전트 실행은 외부 API 활용
- 모니터링은 기존 도구 사용
- 점진적으로 내재화 범위를 확장

---

# Level 5를 마치며

- 하네스 사고의 핵심: 반복 가능 + 관찰 가능 + 개선 가능
- 완벽한 시스템보다 돌아가는 시스템을 먼저, 그 다음 개선
- 이제 에이전트를 '운영'할 수 있는 관점을 갖추었다
- 다음 단계는 이 구조를 팀과 조직에 확산하는 것이다

---

# 실습: 최종 점검

- Stage 1~3에서 만든 산출물을 모두 모은다
- 하네스 스펙, 모니터링 항목, 피드백 루프 설계를 하나로 통합
- 구축/구매 판단을 적용하여 실행 계획을 세운다
- 1주일 안에 첫 번째 버전을 실행해본다
