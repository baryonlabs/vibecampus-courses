# Level 3 · 툴 사용형 AI

AI가 이제 행동하기 시작합니다.

---

# Level 2까지의 한계

- 텍스트 생성만 가능
- 외부 데이터에 접근 불가
- 자동화는 사람이 복사-붙여넣기로 연결
- 실시간 정보 반영이 어려움

---

# 도구를 쓰는 AI란?

- 파일을 읽고 쓸 수 있는 AI
- 검색 엔진을 직접 호출하는 AI
- API를 통해 외부 서비스를 조작하는 AI
- 사람 개입 없이 작업을 수행

---

# 일상 비유로 이해하기

- Level 2: 비서가 메모만 해주는 수준
- Level 3: 비서가 직접 전화하고, 파일 정리하고, 예약까지 해주는 수준
- 핵심 차이 = '행동'이 추가됨

---

# 이 단계에서 하는 일

- 파일 읽기/쓰기
- 검색 결과 정리
- API 호출
- 외부 서비스와 상호작용

---

# 실습 과제

- MCP 서버 목록에서 흥미로운 도구 3개 고르기
- 각 도구가 줄여줄 불편 적기
- 챗봇과 무엇이 다른지 한 줄로 정리

---

<!-- +qa -->
- q: "Level 2와 3의 가장 큰 차이가 뭔가요?"
  a: "Level 2는 AI가 텍스트를 잘 만드는 단계이고, Level 3는 AI가 텍스트 바깥 세계(파일, 웹, API)에 직접 손을 뻗는 단계입니다."
- q: "도구를 쓰면 위험하지 않나요?"
  a: "그래서 권한 관리가 중요합니다. MCP에서는 사용자가 허용한 도구만 AI가 호출할 수 있도록 설계되어 있습니다."

# MCP — Model Context Protocol

AI와 도구를 연결하는 표준 규격

---

# MCP 한 줄 정의

- AI 모델이 외부 도구를 안전하게 호출할 수 있도록 만든 오픈 프로토콜
- Anthropic이 주도하여 공개
- USB처럼 다양한 도구를 하나의 규격으로 연결

---

# MCP의 구성 요소

- MCP Host: AI가 동작하는 앱 (예: Claude Desktop)
- MCP Client: Host 안에서 서버와 통신하는 모듈
- MCP Server: 실제 도구를 제공하는 서비스
- Transport: 호스트와 서버 사이의 통신 방식

---

# MCP가 없던 시절

- 도구마다 연결 방식이 제각각
- 개발자가 매번 커스텀 코드를 작성
- 도구가 바뀌면 전부 수정 필요
- → MCP는 이 문제를 표준화로 해결

---

# MCP가 해결하는 것

- 도구 검색과 등록의 표준화
- 입력/출력 형식 통일
- 권한과 보안 관리
- 재사용 가능한 서버 생태계

---

# 실습 과제

- modelcontextprotocol.io 방문하여 개요 읽기
- MCP 구성 요소 3가지를 자기 말로 정리
- USB 비유가 적절한지 토론해보기

---

<!-- +qa -->
- q: "MCP는 Anthropic 전용인가요?"
  a: "아닙니다. 오픈 프로토콜이라 어떤 AI 모델이든 지원할 수 있습니다. 실제로 다양한 회사가 MCP 서버를 만들고 있습니다."
- q: "MCP 서버를 쓰려면 코딩이 필요한가요?"
  a: "이미 만들어진 MCP 서버는 설정 파일만 수정하면 바로 연결됩니다. 직접 만들 때만 코딩이 필요합니다."

# Tool Calling의 원리

AI가 도구를 '선택'하고 '호출'하는 과정

---

# Tool Calling 흐름

- 1. 사용자가 요청을 보냄
- 2. AI가 적절한 도구를 선택
- 3. AI가 도구에 보낼 입력을 생성
- 4. 도구가 실행되고 결과를 반환
- 5. AI가 결과를 해석하여 답변

---

# AI는 어떻게 도구를 고르나?

- 도구 목록과 설명을 미리 전달받음
- 사용자 요청의 의도를 파악
- 가장 적합한 도구를 JSON 형태로 호출
- 여러 도구를 순서대로 호출할 수도 있음

---

# 실제 예시: 날씨 조회

- 사용자: '서울 날씨 알려줘'
- AI 판단: weather_lookup 도구 필요
- AI 생성: {"location": "Seoul"}
- 도구 반환: {"temp": 18, "condition": "맑음"}
- AI 응답: '서울은 현재 18도이고 맑습니다'

---

# Tool Calling vs 플러그인

- 플러그인: 고정된 기능, 정해진 방식으로만 동작
- Tool Calling: AI가 상황에 맞게 유연하게 도구를 선택
- MCP는 Tool Calling을 표준화한 것

---

# 핵심 정리

- AI는 도구의 '이름과 설명'을 보고 판단한다
- 호출은 구조화된 데이터(JSON)로 이루어진다
- 결과도 구조화되어 AI가 해석할 수 있다

---

# 실습 과제

- Claude에게 '이 파일을 요약해줘'라고 했을 때의 Tool Calling 흐름 그려보기
- 도구 설명(description)이 왜 중요한지 한 문장으로 정리

---

<!-- +qa -->
- q: "AI가 잘못된 도구를 고르면 어떻게 되나요?"
  a: "잘못된 도구를 호출하면 에러가 반환되고, AI가 다시 판단하여 다른 도구를 시도하거나 사용자에게 안내합니다."
- q: "한 번에 여러 도구를 쓸 수 있나요?"
  a: "네, AI는 하나의 요청에서 여러 도구를 순차적으로 또는 병렬로 호출할 수 있습니다. 예를 들어 파일을 읽고 → 요약하고 → 이메일로 보내는 식입니다."
- q: "Tool Calling에 비용이 드나요?"
  a: "도구 호출 자체는 추가 비용이 없지만, AI가 도구 정보를 처리하면서 토큰을 더 사용하게 되어 약간의 비용 증가가 있을 수 있습니다."

# MCP 서버 연결 실습

Claude Desktop에 도구를 붙여봅시다

---

# 연결 전 준비물

- Claude Desktop 앱 설치
- Node.js 또는 Python 환경
- 터미널(명령줄) 기본 사용법
- MCP 서버 패키지 (npm 또는 pip)

---

# 설정 파일 구조

- claude_desktop_config.json 파일 편집
- mcpServers 항목에 서버 추가
- command: 실행할 명령어
- args: 전달할 인자
- env: 필요한 환경 변수 (API 키 등)

---

# 설정 예시: 파일 시스템 서버

```
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/Users/me/Documents"]
    }
  }
}
```

---

# 연결 확인하기

- Claude Desktop 재시작
- 대화창에 도구 아이콘(망치) 확인
- 간단한 요청으로 테스트: '내 Documents 폴더에 어떤 파일이 있어?'
- 도구 호출 승인 팝업이 뜨면 성공

---

# 자주 만나는 문제

- 서버가 시작되지 않음 → 경로와 패키지 확인
- 도구가 목록에 안 뜸 → 설정 파일 JSON 문법 확인
- 권한 에러 → 허용된 디렉토리 경로 확인
- API 키 에러 → env 항목에 키 설정 확인

---

# 실습 과제

- 파일 시스템 MCP 서버를 Claude Desktop에 연결
- Claude에게 특정 폴더의 파일 목록을 물어보기
- 파일 하나를 읽고 요약해달라고 요청해보기

# 도구 파이프라인

여러 도구를 연결하여 복합 작업 수행하기

---

# 파이프라인이란?

- 하나의 도구 결과를 다음 도구의 입력으로 연결
- 사람이 중간에 개입하지 않아도 되는 흐름
- 예: 파일 읽기 → 내용 분석 → 결과를 새 파일로 저장

---

# 실전 시나리오: 회의록 정리

- 1. 파일 서버로 회의 녹취 텍스트 읽기
- 2. AI가 핵심 내용 요약
- 3. 할 일 목록(Action Items) 추출
- 4. 파일 서버로 요약본 저장
- 5. (선택) Slack 서버로 팀에 공유

---

# 실전 시나리오: 경쟁사 조사

- 1. 웹 검색 서버로 최신 뉴스 수집
- 2. AI가 핵심 트렌드 정리
- 3. 파일 서버로 보고서 저장
- 4. 이전 보고서와 비교 분석

---

# 파이프라인 설계 팁

- 각 단계를 작고 명확하게 나누기
- 중간 결과를 확인할 수 있도록 하기
- 에러 발생 시 어디서 실패했는지 알 수 있게
- 처음에는 단계별로 승인하며 진행

---

# 파이프라인의 한계

- 복잡한 분기나 조건 처리가 어려움
- 실패 시 전체를 다시 실행해야 할 수 있음
- 도구 간 데이터 형식이 안 맞을 수 있음
- → 이런 한계는 Level 4의 멀티 에이전트로 해결

---

# 실습 과제

- 2개 이상의 도구를 연결하는 파이프라인 시나리오 설계
- Claude에게 자연어로 파이프라인 작업 요청해보기
- 각 단계에서 어떤 도구가 호출되었는지 기록

---

  a: "현재는 그렇습니다. Level 4에서는 서브 에이전트가 실패를 감지하고 해당 단계만 재시도하는 구조를 배웁니다."

# 나만의 MCP 도구 설계

어떤 도구를 만들어야 할까?

---

# 도구가 필요한 순간 포착하기

- 반복적으로 복사-붙여넣기하는 작업
- 여러 앱을 오가며 하는 작업
- 정해진 순서대로 처리하는 업무
- 데이터를 조회하고 정리하는 일

---

# 좋은 도구의 조건

- 하나의 명확한 기능만 수행
- 입력과 출력이 명확하게 정의됨
- 설명(description)만 읽어도 용도를 알 수 있음
- 에러 상황에 대한 처리가 있음

---

# 도구 설계 3단계

- 1. 이름 짓기: 동사_명사 형식 (예: search_emails)
- 2. 입력 정의: 어떤 정보가 필요한가 (예: query, date_range)
- 3. 출력 정의: 어떤 형태로 결과를 줄 것인가 (예: 이메일 목록)

---

# 설계 연습: 사내 위키 검색 도구

- 이름: search_wiki
- 입력: {query: string, category?: string}
- 출력: {results: [{title, url, snippet}]}
- 설명: '사내 위키에서 키워드로 문서를 검색합니다'

---

# 흔한 설계 실수

- 도구 하나에 너무 많은 기능을 넣기
- 설명이 모호해서 AI가 언제 써야 할지 모름
- 입력 검증이 없어서 에러가 많음
- 출력 형식이 불일치하여 후속 처리가 어려움

---

# 실습 과제

- 내 업무에서 도구로 만들면 좋을 작업 3가지 나열
- 그 중 하나를 이름/입력/출력/설명 형식으로 설계
- 동료에게 설명만 보여주고 용도를 맞추는지 테스트

---

<!-- +qa -->
- q: "코딩을 못해도 도구를 설계할 수 있나요?"
  a: "설계는 코딩과 다릅니다. '어떤 입력을 받아서 어떤 결과를 주는가'를 정의하는 것이 핵심이고, 구현은 개발자나 AI에게 맡길 수 있습니다."
- q: "도구 설명이 왜 그렇게 중요한가요?"
  a: "AI는 도구의 설명을 읽고 '이 도구를 지금 써야 하는지'를 판단합니다. 설명이 모호하면 필요할 때 안 쓰거나, 불필요할 때 쓰게 됩니다."

# Level 3의 벽

도구만으로는 풀 수 없는 문제들

---

# Level 3의 성과 정리

- AI가 외부 세계와 상호작용 가능
- 반복 작업의 자동화
- 여러 도구의 파이프라인 구성
- MCP 생태계를 통한 빠른 확장

---

# 한계 1: 단일 AI의 인지 부하

- 도구가 많아지면 AI가 선택에 혼란
- 긴 작업에서 맥락을 잃기 쉬움
- 하나의 AI가 모든 역할을 수행하기 어려움

---

# 한계 2: 복잡한 작업 분리 불가

- 리서치 + 분석 + 작성을 한 AI가 전부 담당
- 각 단계별 전문성을 갖추기 어려움
- 실패 시 어느 단계에서 문제인지 파악 곤란

---

# 한계 3: 자율성의 부족

- 매 단계마다 사람의 승인이 필요
- 예상치 못한 상황에 스스로 대처 어려움
- 계획을 세우고 조정하는 능력이 부족

---

# Level 4가 필요한 이유

- 여러 AI가 역할을 나눠 협업하는 구조
- 서브 에이전트가 각각 전문 도구를 사용
- 오케스트레이터가 전체를 조율
- 실패한 부분만 재시도 가능
- → 팀처럼 일하는 AI 시스템

---

# Level 3 졸업 체크리스트

- MCP의 개념과 구조를 설명할 수 있다
- MCP 서버를 찾고 연결할 수 있다
- 도구 파이프라인을 설계할 수 있다
- 커스텀 도구를 설계(이름/입력/출력)할 수 있다
- Level 3의 한계를 명확히 인식한다

---

  a: "물론입니다. 단순하고 정형화된 업무 자동화는 Level 3만으로 충분합니다. 복잡한 프로젝트 관리나 다단계 의사결정이 필요할 때 Level 4를 고려하세요."
